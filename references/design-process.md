## 一、设计的基本立场

产品设计不是把功能放进页面，也不是把页面做得更“现代”。设计的任务是：在特定上下文中，让特定用户以可理解、可控制、可恢复的方式达成目标，并让系统、组织和业务能够长期承受这一方案。

设计工作应始终回答六个问题：

1. 谁在什么情境下试图完成什么？
2. 用户此刻感受到什么，又害怕发生什么？
3. 当前系统在哪些地方与真实情境不匹配？
4. 问题发生在产品、信息架构、功能还是组件层？
5. 方案引入了哪些新风险、依赖和认知负担？
6. 有什么证据能区分事实、推断与建议？

一个可取的方案不只追求完成效率，还应同时维护：

- **可理解性**：用户知道当前状态、下一步和动作后果。
- **可控性**：用户能限制范围、修改计划、暂停、取消和撤销。
- **连续性**：信息与工作可以跨时间、成员和设备延续。
- **可恢复性**：失败不会把用户留在不可逆或不可解释的状态。
- **一致性**：相同概念、动作与风险使用稳定的表达和行为。
- **适应性**：方案能容纳真实世界的例外，而非只服务理想主路径。

## 二、Chair、Car、City：先判断问题尺度

Chair、Car、City 是判断设计问题尺度的比喻。尺度不同，研究方法、决策权限、验证方式与交付物也不同。

### 2.1 Chair：单点对象与局部交互

Chair 尺度关注一个边界明确、可独立理解的对象，例如按钮、属性编辑器、权限弹窗、数据库单元格或页面发布控件。

典型问题：

- 控件是否容易发现和理解？
- 状态与反馈是否明确？
- 是否支持键盘、触控和无障碍使用？
- 错误是否可预防、可恢复？
- 组件是否与设计系统一致？

主要交付物：组件规格、交互状态、文案、可访问性要求、埋点和验收标准。

常见误区：用局部视觉优化掩盖上层流程或信息架构问题。

### 2.2 Car：跨功能任务与完整旅程

Car 尺度关注多个组件协作形成的端到端任务，例如“从会议记录生成项目页面”“筛选数据库后批量更新状态”“创建并启用一个自动化”。

典型问题：

- 用户能否建立正确的任务心智模型？
- 各步骤如何衔接，信息如何在步骤间传递？
- 何时需要确认，何时应自动推进？
- 中断、失败、冲突和恢复如何处理？
- 不同功能是否围绕同一个用户目标协同？

主要交付物：用户旅程、服务蓝图、流程图、状态矩阵、权限与风险策略、端到端原型。

常见误区：把完整任务拆成若干“局部都合理”的页面，却没有人对整体完成率和中断成本负责。

### 2.3 City：生态、治理与长期演化

City 尺度关注平台、工作空间或组织级系统，例如团队知识库、数据库生态、Agent 权限治理、多角色协作和跨工具自动化。

典型问题：

- 对象、角色、权限和空间之间的关系是什么？
- 谁拥有规则，谁承担错误后果？
- 局部选择如何影响全局一致性和未来扩展？
- 系统怎样支持不同团队形成自己的工作方式，同时避免失控？
- 数据、自动化与组织变动如何长期演化？

主要交付物：产品原则、对象模型、信息架构、治理模型、权限体系、平台能力地图、迁移策略和演进路线图。

常见误区：用一个“万能首页”或更多设置项解决本质上的对象模型和治理问题。

### 2.4 尺度判断规则

评审前先写明主尺度，并检查相邻尺度：

- 一个控件问题反复出现，通常意味着 Car 或 City 层存在结构性问题。
- 一个流程需要大量解释，可能不是文案问题，而是信息架构或对象模型问题。
- 一个平台方案无法落地，可能需要下沉到关键旅程和组件状态验证。
- 方案可以跨尺度，但不能用 Chair 级证据证明 City 级结论。

## 三、Christopher Alexander 的 fit 思想

Christopher Alexander 将设计理解为形态（form）与上下文（context）之间不断取得良好适配（fit）的过程。设计问题不是“还缺哪个功能”，而是当前形态在哪些方面与真实情境失配（misfit）。

### 3.1 用失配陈述替代模糊需求

不写：

> 用户需要一个更智能的数据库助手。

改写为可观察的失配：

> 当用户要求 Agent 批量修改数据库时，系统没有在执行前呈现对象范围与字段差异，导致用户无法判断影响，也不敢授权执行。

一个好的失配陈述包含：

- 上下文：何时、何地、由谁使用。
- 目标：用户试图达成什么。
- 当前形态：系统目前如何表现。
- 冲突：当前形态为何不适配。
- 后果：造成时间、错误、风险或情绪上的什么损失。
- 证据：观察、数据、日志、访谈或明确约束。

### 3.2 Fit 不是一次性最优解

上下文会随着团队规模、数据结构、权限规则和用户熟练度变化。方案应被视为“当前约束下的适配”，而不是永恒正确的答案。因此需要：

- 明确适用边界和前提。
- 记录仍未解决的失配。
- 为高不确定部分设计可逆实验。
- 通过运行数据继续发现新失配。
- 避免为了局部适配制造更严重的系统性失配。

### 3.3 Fit 评审问题

- 方案适配的是实际行为，还是理想化流程？
- 它解决了哪个明确失配？
- 它将成本转移给了谁？
- 它在哪些边界条件下会失效？
- 它是否引入新的认知、权限、数据或运营失配？
- 当上下文变化时，方案能否安全退化或调整？

## 四、产品、信息架构、功能、组件四层

设计决策应标注所属层级，并从上到下保持一致。

### 4.1 产品层：为何存在，向谁负责

产品层定义价值、边界和行为原则。

需要回答：

- 核心用户与关键情境是什么？
- 产品承诺帮助用户完成什么？
- 哪些事情产品明确不做？
- 自主、控制、速度、安全之间如何取舍？
- 成功由什么用户结果衡量，而非只看使用量？

输出：产品原则、目标用户、价值主张、非目标、核心对象、成功指标和风险边界。

### 4.2 信息架构层：世界由什么组成

信息架构定义对象、关系、层级、命名、导航和归属。

需要回答：

- 系统中的一等对象是什么，例如页面、数据库、任务、自动化、权限和日志？
- 对象之间如何创建、引用、嵌套、移动、共享和归档？
- 用户去哪里查看、继续或管理一项工作？
- 相同信息是否在不同入口保持一致？
- 不同角色看到的是同一对象的不同视图，还是不同对象？

输出：对象模型、关系图、导航结构、内容分类、命名规范和生命周期。

### 4.3 功能层：用户如何完成任务

功能层定义端到端能力、业务规则和状态变化。

需要回答：

- 触发条件、输入、处理、输出是什么？
- 主路径和例外路径是什么？
- 哪些步骤自动进行，哪些需要用户判断？
- 权限、确认、通知、失败和撤销怎样进入流程？
- 功能是否形成闭环，还是把剩余工作丢给用户？

输出：任务流、业务规则、状态机、权限策略、异常处理和验收标准。

### 4.4 组件层：每个接触点如何工作

组件层定义控件、布局、视觉、文案、反馈和无障碍。

需要回答：

- 默认、悬停、聚焦、禁用、加载、成功和错误状态是否完整？
- 可点击区域、层级、对比度和键盘路径是否合理？
- 文案是否说明动作与后果？
- 组件是否符合平台惯例和设计系统？
- 高风险动作是否具备足够但不过度的摩擦？

输出：界面规格、组件状态、交互文案、响应式行为和无障碍要求。

### 4.5 四层诊断原则

- 先判断问题最高发生层，再向下落实；不要从组件层逆向拼产品策略。
- 上层决定“为什么与什么”，下层决定“怎样呈现与操作”。
- 上层问题若未解决，下层精细化只会让错误方向更昂贵。
- 评审意见必须标注层级，避免把战略争议伪装成按钮位置争议。

## 五、从上下文与用户感受出发的完整设计流程

### 第 0 步：建立任务契约

明确：

- 要解决的决策，而不只是要产出的页面。
- 主要用户、业务目标与时间范围。
- 已知约束、不可触碰边界和决策人。
- 所需证据、交付物和成功判定。
- 当前问题尺度与涉及层级。

### 第 1 步：进入上下文

收集真实使用场景，而非只接受抽象需求：

- 用户在任务发生前做了什么？
- 触发任务的事件是什么？
- 使用了哪些文档、数据库、消息或外部工具？
- 有哪些时间压力、协作关系和权限限制？
- 任务完成后，结果由谁继续使用？
- 当前替代方案和人工补丁是什么？

优先使用行为证据：屏幕观察、任务回放、日志、现有内容和真实案例。访谈中的观点用于解释行为，不能替代行为本身。

### 第 2 步：描绘用户感受

感受不是装饰性“同理心”，而是影响决策和信任的设计输入。记录关键节点的：

- 期待：用户希望系统替自己承担什么？
- 不确定：用户不知道哪些信息或后果？
- 焦虑：用户最担心出错、丢失或越权的是什么？
- 负担：哪些判断、记忆和重复劳动落在用户身上？
- 掌控感：哪些时刻用户需要介入或确认？
- 成就感：什么结果会让用户相信任务真正完成？

不要只写“用户感到困惑”。应连接原因与行为：

> 用户无法看到批量修改范围，因此反复缩小指令并手工抽查；这不是对 Agent 能力的不信任，而是缺少可验证的影响预览。

### 第 3 步：建立事实地图与证据台账

整理：

- 已确认事实
- 尚待验证的推断
- 组织或技术约束
- 用户差异与例外
- 已知数据缺口
- 相互矛盾的证据

为每条关键判断标注来源、时间和可信边界。证据不足时，先降低结论强度，而非用更多措辞填补空白。

### 第 4 步：列出失配，而非立即画方案

将问题拆成具体失配，并按影响、频率、风险和可干预程度排序。

失配格式：

> 在【上下文】中，【用户】试图【目标】，但当前【系统形态】导致【冲突】，因此产生【后果】。证据为【来源】。

检查失配之间是否存在因果关系，识别根因与表象。例如“按钮找不到”可能来自对象归属错误，而不是按钮视觉权重不足。

### 第 5 步：确定设计原则与取舍

针对本项目写 3—5 条可用于做决定的原则。原则必须能排除方案，例如：

- 写操作先展示影响，再要求承诺。
- 低风险动作减少确认，高风险动作增加证据与恢复能力。
- 不要求用户理解系统内部工具边界，按用户任务组织流程。
- 自动化必须拥有可见状态、明确所有者和停止方式。

同时记录关键取舍：速度与控制、灵活与一致、个体效率与组织治理、即时价值与长期维护。

### 第 6 步：探索多条方案路线

至少提出三类路线，而不是同一布局的视觉变体：

- 保守路线：最小改变，低迁移成本。
- 平衡路线：重组关键旅程，保留现有心智模型。
- 结构路线：调整对象模型、信息架构或治理方式。

每条路线应说明：解决哪些失配、未解决什么、引入什么风险、依赖什么前提，以及为何适合当前尺度。

### 第 7 步：维护 Wilderness

将被否方案、未验证想法、边界案例和暂缓问题放入 Wilderness，具体规则见下一章。它不是垃圾桶，而是防止团队反复重走旧路、过早收敛或遗忘重要风险的外部记忆。

### 第 8 步：建立信息架构与任务骨架

先定义对象、关系、入口、归属和状态，再决定页面布局。检查：

- 用户从哪里开始？
- 工作结果保存在哪里？
- 中断后从哪里继续？
- 谁拥有、管理和审计？
- 相同对象在不同场景如何保持一致？

### 第 9 步：完成端到端流程与状态矩阵

覆盖主路径、权限、确认、空状态、加载、部分成功、冲突、取消、失败、重试、撤销和过期。不要只原型化成功路径。

### 第 10 步：风险建模与视觉层级

为每个关键动作评估影响、可逆性、范围、概率、可检测性和权限敏感度，并据此决定视觉权重、确认摩擦与恢复机制。

### 第 11 步：原型与验证

按最大不确定性选择验证方式：

- 概念不确定：访谈、故事板、概念测试。
- IA 不确定：卡片分类、树测试、对象理解测试。
- 流程不确定：任务原型、可用性测试、服务走查。
- 风险不确定：失败演练、权限测试、撤销测试。
- 视觉不确定：五秒测试、可发现性与层级测试。
- 技术不确定：可行性原型、日志验证、影子运行。

验证任务应使用真实数据和真实约束，至少包含一个失败或冲突场景。

### 第 12 步：决策、交付与后续学习

交付不只包含最终界面，还应包含：

- 决策摘要与证据
- 适用范围和非目标
- 四层设计说明
- 状态矩阵与风险策略
- 被否方案及否决理由
- 埋点、成功指标和监控阈值
- 上线后的回滚与学习计划

## 六、Wilderness：保留被否方案与未决空间

Wilderness 是设计过程中的“未被选中但仍有价值的空间”。它包括：

- 被否方案及其理由
- 暂时不具备前提的方案
- 低频但高风险的边界案例
- 证据不足的假设
- 尚未解决的失配
- 因时间或范围被推迟的问题
- 可能在下一阶段复用的组件或机制

### 6.1 为什么需要 Wilderness

- 防止同一失败方案在新成员加入后被重复提出。
- 保留少数意见与非主流场景，避免过早平均化。
- 区分“错误”与“当前不适合”。
- 让方案的边界和代价可见。
- 当约束变化时，可以快速重新评估旧路线。

### 6.2 Wilderness 记录格式

| 字段 | 内容 |
|---|---|
| 名称 | 方案或未决问题的短名称 |
| 类型 | 被否方案／待验证假设／边界案例／延期问题 |
| 原始目标 | 它试图解决什么失配 |
| 核心做法 | 方案如何运作 |
| 当前结论 | 否决、暂缓、保留观察或待验证 |
| 理由 | 证据、风险、成本或依赖 |
| 重新开启条件 | 什么变化后值得重评 |
| 负责人 | 谁维护该项判断 |
| 日期 | 最近一次评估时间 |

### 6.3 被否方案的表达纪律

不要写“体验不好”“太复杂”。应写可检验的理由：

> 否决全局悬浮确认条。它在批量编辑测试中遮挡了当前记录上下文，且用户无法将确认动作与具体变更集对应。若未来引入稳定的任务对象和变更集 ID，可重新评估。

## 七、状态矩阵

状态矩阵用于确保方案覆盖系统真实运行，而非只展示一条理想流程。

### 7.1 通用状态矩阵

| 状态 | 系统发生什么 | 用户需要知道什么 | 主要动作 | 恢复／后续 | 日志要求 |
|---|---|---|---|---|---|
| 初始 | 尚未开始 | 能做什么、需要什么输入 | 开始、查看示例 | 返回上级 | 可选 |
| 空状态 | 没有数据或结果 | 为什么为空、如何产生内容 | 创建、连接、刷新 | 保留上下文 | 记录关键空因 |
| 输入中 | 用户正在配置 | 当前范围与约束 | 编辑、取消、继续 | 自动保存 | 记录草稿版本 |
| 校验中 | 检查输入和权限 | 正在检查什么 | 等待、停止 | 返回可修正位置 | 记录校验结果 |
| 计划就绪 | 已形成执行计划 | 步骤、影响、风险、来源 | 修改、批准、取消 | 保存为草稿 | 记录计划版本 |
| 等待权限 | 权限不足 | 缺什么、为何需要、授权多久 | 授权、缩小范围、拒绝 | 转只读或草稿 | 记录授权依据 |
| 等待确认 | 高风险动作未执行 | 将改变什么、是否可撤销 | 确认、排除、编辑 | 回到计划 | 记录确认人和范围 |
| 执行中 | 系统正在处理 | 当前步骤、已完成项、剩余范围 | 暂停、停止 | 安全中断 | 记录每个写动作 |
| 已暂停 | 执行暂时停止 | 停止点和已发生变更 | 继续、取消、回滚 | 校验数据新鲜度 | 记录暂停原因 |
| 部分成功 | 部分动作失败或跳过 | 成功、失败、未执行各多少 | 重试、保留、回滚 | 分项恢复 | 逐项记录结果 |
| 已成功 | 目标已完成并验证 | 结果、变更对象、异常 | 查看、继续、撤销 | 进入结果对象 | 保存结果摘要 |
| 已失败 | 无法继续 | 原因、是否产生副作用 | 重试、修改、求助 | 回滚已完成部分 | 保存错误与上下文 |
| 已取消 | 用户或系统停止 | 已完成到哪里 | 恢复、撤销、关闭 | 清理临时状态 | 记录取消主体 |
| 冲突 | 数据已被其他人或流程修改 | 冲突对象与差异 | 合并、跳过、覆盖 | 重新验证 | 记录版本与选择 |
| 权限过期 | 运行身份失效 | 受影响任务和数据 | 重新授权、转移所有权 | 暂停自动化 | 记录权限变化 |
| 已撤销 | 变更被恢复 | 恢复范围和未恢复项 | 查看、重新执行 | 保留审计链 | 链接原变更集 |
| 不可逆 | 动作无法完全恢复 | 明确后果和替代补救 | 最终确认、取消 | 补偿动作 | 强制审计 |

### 7.2 状态矩阵的交叉维度

针对关键流程，还要逐一交叉检查：

- 数据：无数据、少量、大量、过期、冲突、敏感。
- 权限：完整、只读、部分、临时、过期、被撤回。
- 网络：正常、慢、离线、超时、重复提交。
- 协作：单人、多人同时编辑、所有者离职、对象被移动。
- 设备与输入：桌面、移动、键盘、触控、辅助技术。
- 运行方式：人工触发、定时触发、事件触发、重试。
- 结果：成功、部分成功、失败、取消、撤销失败。

## 八、风险与视觉权重

视觉权重不等于业务重要性。界面应优先突出用户此刻必须理解和决定的内容，并与真实风险匹配。

### 8.1 风险评估维度

对动作按以下维度评估：

- **影响强度**：错误会造成多大损失？
- **影响范围**：单个对象、批量对象、整个空间还是外部系统？
- **可逆性**：能否完整、即时、低成本撤销？
- **发生概率**：在真实条件下多容易出错？
- **可检测性**：错误能否在造成后果前被发现？
- **权限敏感度**：是否涉及敏感数据、成员、分享或管理权限？
- **时间敏感度**：是否会立即发布、发送或触发下游流程？
- **自动化程度**：单次人工动作还是持续无人值守运行？

### 8.2 风险等级与交互策略

| 风险 | 示例 | 视觉权重 | 交互策略 | 恢复要求 |
|---|---|---|---|---|
| 低 | 读取页面、创建私人草稿 | 轻量 | 可直接执行，完成后反馈 | 普通撤销或删除 |
| 中 | 修改单条记录、移动页面 | 明确但不警报化 | 展示目标与结果，必要时轻确认 | 单动作撤销 |
| 高 | 批量覆盖、公开发布、启用自动化 | 高 | 影响预览、范围确认、明确主语和后果 | 变更集、暂停、回滚 |
| 极高 | 永久删除、权限变更、外部发送、跨空间敏感数据传输 | 最高且克制 | 分步确认、重新认证或双人审批；禁止默认选择 | 补偿方案、强制审计 |

### 8.3 视觉权重原则

- 当前状态和下一项关键决定优先于品牌装饰。
- 风险信息靠近触发动作，不埋在说明页或二级弹窗。
- 危险色只表达真实危险，不用于普通取消、返回或品牌强调。
- 主按钮代表用户最可能且安全的下一步，不代表产品最希望用户点击的动作。
- 高风险动作使用清楚的动词和对象，如“删除 128 条记录”，不用“继续”。
- 低风险频繁动作减少警告，避免确认疲劳削弱真正警告的效果。
- 成功反馈应说明结果和去向，不只显示绿色对勾。
- 执行中的视觉进度应体现语义步骤和实际范围，不使用虚假的精确百分比。

### 8.4 风险与证据一起呈现

高风险决策应同时提高证据可见性：变更前后差异、对象数量、权限依据、来源、可逆性和异常项。风险越高，用户越不应仅凭 Agent 的自然语言自信做决定。

## 九、事实、推断、建议的证据协议

所有评审与方案必须区分事实（Fact）、推断（Inference）和建议（Recommendation），避免把观点包装成证据。

### 9.1 三类陈述

#### 事实 F

可以由明确来源直接支持的观察、数据、约束或已实现行为。

格式：

> **[F]** 过去 30 天有 42% 的批量编辑任务在确认页被取消。来源：任务日志，2026-06-01 至 2026-06-30，样本 3,412 次。

要求：来源、时间、样本或适用范围、必要的限制说明。

#### 推断 I

基于一个或多个事实做出的解释、因果假设或预测。

格式：

> **[I]** 用户可能不是放弃任务，而是因确认页缺少字段级差异而无法判断风险。依据：[F1] 取消率；[F2] 访谈中 6/8 人要求先看差异。待通过 diff 原型测试验证。

要求：引用依据、说明不确定性、列出替代解释和验证方式。

#### 建议 R

基于事实、推断、设计原则和约束提出的行动方案。

格式：

> **[R]** 在批量修改确认页增加字段级 diff，并允许排除单条记录。依据：[F1][F2][I1]；目标是降低不确定性，不承诺一定降低取消率。优先通过 20 人任务测试验证。

要求：依据、目标、适用范围、代价、风险和验证标准。

### 9.2 证据等级

可按任务需要使用以下等级，不把它当作机械评分：

- **E0 无证据**：灵感或主张，只能进入 Wilderness 或待验证假设。
- **E1 间接证据**：行业惯例、类比、专家判断。
- **E2 定性直接证据**：真实用户观察、访谈、工单、任务回放。
- **E3 定量直接证据**：产品日志、实验、结构化调查。
- **E4 多源一致证据**：定性与定量相互支持，且覆盖关键人群与边界。

证据等级表示当前支持强度，不代表高等级证据自动产生正确方案。City 级决策通常需要多源证据；Chair 级小改动可以通过低成本实验迅速验证。

### 9.3 证据台账模板

| ID | 类型 | 陈述 | 来源／依据 | 时间与范围 | 可信边界 | 下一步 |
|---|---|---|---|---|---|---|
| F1 | 事实 |  |  |  |  |  |
| I1 | 推断 |  | F1、F2 |  |  | 验证方式 |
| R1 | 建议 |  | F1、I1、原则 P1 |  | 代价与风险 | 决策或实验 |

### 9.4 表达规则

- 不将“用户表示喜欢”写成“用户会使用”。
- 不将相关性写成因果关系。
- 不把内部利益相关者意见称为用户需求。
- 不用单个案例代表总体，但高风险案例可作为风险证据。
- 信息缺失时写“尚未确认”，不编造完整叙事。
- 方案建议必须能追溯到失配、证据或明确原则。
- 新证据推翻旧判断时，更新结论并保留决策历史。

## 十二、交付前检查清单

### 问题与证据

- [ ] 已写明用户、上下文、目标和后果，而不是只有功能需求。
- [ ] 已判断 Chair、Car、City 主尺度及相邻尺度影响。
- [ ] 已用 fit／misfit 描述核心问题。
- [ ] 已明确区分事实、推断和建议。
- [ ] 每个关键建议可追溯到证据、失配或设计原则。
- [ ] 已记录证据缺口与替代解释。

### 产品与结构

- [ ] 产品、信息架构、功能、组件四层保持一致。
- [ ] 核心对象、关系、归属、入口和生命周期明确。
- [ ] 方案按用户任务而非内部系统结构组织。
- [ ] 已说明非目标和适用边界。

### 交互与状态

- [ ] 主路径从触发到结果形成闭环。
- [ ] 状态矩阵覆盖空、加载、权限、确认、执行、成功、部分成功、失败、取消、冲突与撤销。
- [ ] 用户始终知道系统在做什么、为什么能做、如何停止或恢复。
- [ ] 自动化有所有者、运行状态、异常策略和停止入口。

### 风险与视觉

- [ ] 确认摩擦与真实风险匹配。
- [ ] 高风险动作明确展示对象、范围、后果和可逆性。
- [ ] 视觉权重服务于当前决策，不被装饰或低价值信息占用。
- [ ] 危险色与警告没有被滥用。
- [ ] 已覆盖敏感数据、权限变化和不可逆动作。

### 学习与演进

- [ ] 已保留被否方案、未决问题和重新开启条件。
- [ ] 已针对最大不确定性设计验证，而非只验证偏好。
- [ ] 已定义成功、停止、回滚和复盘标准。
- [ ] 上线后能通过日志和指标发现新的失配。

这套流程的目的不是增加文档负担，而是在正确尺度上做出可追溯、可验证、可恢复的设计决策。简单问题可以只使用相关章节；跨功能或平台级问题应完整覆盖四层、状态、风险、证据与 Wilderness。
