## How to use this index

Use the smallest source set that can establish the claim:

1. For **current behavior**, start with current Notion Help or release notes.
2. For **launch intent**, use the dated launch article or release entry.
3. For **design rationale**, use a first-person design or engineering article and the Design Matters talk.
4. For **mission and history**, use founder interviews and investor profiles, then mark broad interpretations as synthesis.
5. For **guidance**, cite the Notion evidence first and make the transfer to another product explicit as an inference.

Do not use a historical launch article as proof that the interface or permission behavior is unchanged today.
Do not treat this index's URL, summary, or a synthesized reference as evidence that the linked page was opened in the current task. Open the primary page before using **Fact (current)**. If that is impossible, report the source date and say it was not rechecked today.

## Source authority and freshness

| Source type | Best for | Freshness rule |
|---|---|---|
| Current Help Center | Present interaction, terminology, limits | Recheck online for any implementation claim |
| Dated release note | What launched and when | Treat as historical behavior |
| Design/engineering blog | Rationale, alternatives, mechanisms | Separate intent from current UI |
| First-person talk/interview | Process, philosophy, organizational context | Avoid converting recollection into precise product fact |
| Third-party profile | Independent chronology and quotations | Use primary source when available for interaction details |
| Local transcript | Exact wording and sequence in the 2022 talk | Cite timestamp; check official page for attribution |

## Citation patterns

### Current behavior

> **Fact (current Help Center, checked YYYY-MM-DD):** Users create a synced block through …

Then cite the Help Center URL. If comparing with launch behavior, add a separate historical sentence with the dated launch source.

### Historical design intent

> **Fact (2021 launch/design article):** The original interaction used familiar copy and paste as the entry into syncing.

Do not imply every visual detail remains current.

### Transferable guidance

> **Inference from the Synced Blocks case:** Preserve the normal meaning of copy; reveal shared identity at the explicit “sync” choice and again before editing affects other locations.

This wording makes the evidence-to-guidance bridge visible.

### Weak or conflicting evidence

> Current permission inheritance could not be established from a sufficiently precise primary source. Treat the following as a design question, not a description of Notion's current behavior.

## Known evidence gaps

- There is no sufficiently comprehensive first-party source in this collection proving that Notion meets a particular accessibility standard. Audit keyboard, screen-reader, contrast, focus, motion, zoom, and touch-target behavior independently.
- Current permission behavior for cross-page, cross-teamspace, guest, and synced content can change. Verify it in current Help or the product before making a factual claim.
- Launch posts emphasize benefits and intent; they do not establish adoption, user satisfaction, or comparative superiority.
- Product terminology and UI copy change. Quote exact copy only after checking the current product or Help Center.
- “LEGO-like” is a useful description of composability, but do not present it as a verbatim founder quote unless the exact source confirms the words.
