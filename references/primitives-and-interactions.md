# Notion 式原语与交互参考

本文用于指导类 Notion 工作软件的产品架构与交互设计。重点不是复刻某个界面，而是保持一套可组合、可预测、可撤销的对象模型与交互语法，使空白画布、模板、数据库和团队工作流共享同一底层逻辑。

## 一、设计原则

1. **一套原语，多种表达。** 页面、数据库记录、模板实例都应由相同的块和属性系统组成，避免空白画布与预制工作流成为两套产品。
2. **直接操作优先。** 用户看到的内容就是操作对象；点击编辑、拖拽移动、就地改名，减少“配置后才能看到结果”的间接感。
3. **先给低门槛，再开放高上限。** 首屏只暴露高频动作，高级结构在意图出现时通过命令、上下文菜单或属性面板显现。
4. **保持对象身份。** 移动、转化、切换视图时尽量保留对象 ID、内容、引用、评论和历史，避免“看似变换，实际复制”的断裂。
5. **操作可预测、可逆。** 在执行前说明范围，在执行后即时反馈并提供撤销；破坏性操作必须区分“移除呈现”和“删除源数据”。
6. **协作边界可见。** 权限、共享范围、同步状态和他人编辑状态不应隐藏在错误发生之后。

## 二、核心对象模型

### Block

Block 是最小可组合内容单元，也是画布的统一操作对象。

建议字段：

- `id`：稳定身份，用于引用、评论、历史和同步。
- `type`：段落、标题、待办、列表、图片、嵌入、数据库视图等。
- `content`：文本或类型特定载荷。
- `attributes`：颜色、对齐、状态、尺寸等可选配置。
- `children`：子块列表，表达嵌套关系。
- `parent` 与 `position`：归属和同级顺序。
- `permissions`、`created_by`、`updated_at`：协作与审计信息。

设计要求：

- 新建块必须足够轻：回车即可产生下一个块，不要求先选类型。
- 块类型可以转换，但转换契约必须明确哪些内容会保留、映射或丢失。
- 块可以选择、复制、移动、缩进、引用和删除，所有操作遵循同一套选择语义。
- 空块既是输入位置，也是插入入口；不要让空状态充满永久工具栏。
- 复杂块内部可有自己的编辑模式，但退出后仍回到统一块模型。

### Page

Page 是具有稳定身份的容器，也是导航、分享、权限、搜索和链接的主要边界。可将其理解为一种拥有标题与元数据的特殊块。

建议组成：

- 页面身份：ID、标题、图标、封面、创建者、更新时间。
- 页面内容：按顺序排列的子块树。
- 页面属性：当页面属于数据库时，承载该数据库定义的字段值。
- 页面关系：父页面、所属数据库、提及、反向链接、评论和权限继承。

设计要求：

- 普通页面与数据库记录页应共享编辑体验，差异主要体现在属性区和来源上下文。
- 页面移动改变位置而非身份，原链接、反向链接和评论应继续有效。
- 面包屑表达当前位置，但不能成为唯一返回路径。
- 权限继承需可解释；当页面权限与父级不同时，应显示明确提示。

### Database

Database 是“结构化页面集合 + 属性模式”，不是某一种表格界面。每条记录本质上是一个页面，表格、看板、日历等只是观察和操作这些页面的不同方式。

建议组成：

- 数据源身份：名称、ID、权限和所有者。
- Schema：属性名称、类型、约束、默认值和选项。
- Records：页面集合，每个页面保存属性值与正文块。
- Views：针对同一数据源的多个投影配置。
- Relations/Rollups：跨数据源关联与派生结果。

设计要求：

- 修改视图的筛选、排序、分组和列宽，不应复制或改写源记录。
- 删除视图与删除数据库必须是两个清晰不同的动作。
- 属性类型变更要预览不可逆影响，例如文本转日期可能造成无法解析的数据。
- 内嵌数据库与整页数据库可共享数据模型，只改变容器和导航呈现。
- 链接数据源应显示来源、权限和同步关系，避免用户误以为它是副本。

### View

View 是数据库的可保存投影与交互布局。它回答“以什么方式看和操作同一批数据”，不拥有记录本身。

一个视图通常包含：

- 布局：表格、列表、看板、日历、时间线、画廊等。
- 查询：筛选、排序、分组、搜索条件。
- 呈现：可见属性、属性顺序、列宽、卡片预览、密度。
- 视图级偏好：打开方式、默认模板、是否显示统计。

设计要求：

- 视图切换应保留用户所在上下文；必要时保留选中记录或滚动锚点。
- 视图配置需要明确“仅自己”还是“对所有协作者”，防止无意打乱团队界面。
- 空视图必须区分：源数据为空、筛选后无结果、无权限、加载失败。
- 对不兼容布局的配置进行可解释迁移，而不是静默丢弃。

### Template

Template 是创建对象时使用的初始化配方，可包含默认属性、块树、说明、视图和自动化入口。模板不是实例的永久父级，除非产品明确提供“同步模板”能力。

设计要求：

- 应用前预览将创建的内容、属性、视图和可能影响的范围。
- 示例内容、结构内容和自动化应有明确标识。
- 实例创建后可自由编辑；模板后续变化不应静默覆盖实例。
- 支持模块化采用：用户可只添加某个视图、页面结构或自动化，而非被迫接受整套方案。
- “清空示例但保留结构”应成为一等操作。
- 团队模板需要来源、维护者、版本或认证状态；个人模板与官方模板不可混淆。

### 对象关系与不变量

应始终保持以下不变量：

- Page 包含 Block；Block 可以继续包含子 Block。
- Database 定义 schema 并聚合记录页；记录页仍可包含任意 Block。
- View 引用 Database，不拥有记录。
- Template 创建或初始化上述对象，不默认持续控制实例。
- Mention 指向稳定对象 ID，而不是只保存显示名称。
- 移动改变父级和位置；复制才创建新身份。
- 视图级隐藏不等于数据删除；权限不可见不等于数据不存在。

## 三、交互语法

### Slash 命令

Slash 是键盘优先的插入与转换入口，适合在编辑流中表达“此处加入什么”。

- 输入 `/` 打开与光标位置绑定的命令面板。
- 支持按名称、别名和常见意图搜索，如“待办”“图片”“表格”。
- 默认排序结合高频命令、当前上下文和最近使用，但不能让核心命令位置剧烈漂移。
- 上下键移动、回车确认、Esc 关闭；关闭后恢复原文本与光标。
- 零结果状态提供纠错、相近命令或继续输入为普通文本的路径。
- 仅显示当前位置可用的命令；不可用项若对理解重要，应保留并解释原因。
- 触屏端可用“+”或快捷工具条承接相同能力，不强迫用户输入 `/`。

### 六点菜单与块句柄

六点句柄同时承担“这是一个可移动对象”的视觉暗示和块级操作入口。

- Rest 状态尽量隐藏；指针进入块的命中区后显示。
- 点击打开块菜单，按下并移动进入拖拽；两者应有明确的位移阈值，防止误触。
- 菜单包含高频块操作：删除、复制、转化、颜色、链接、评论等。
- 多选时菜单作用于选择集，并明确数量与不兼容操作。
- 句柄命中区可以大于图标本身，保证易点按，但不得遮挡正文选择。
- 触屏端以长按或显式更多按钮替代 hover 发现。

### 拖拽

拖拽是移动而非复制的默认语义，必须持续回答三个问题：正在移动什么、会放到哪里、会形成什么层级。

- 拖起后显示对象数量与轻量拖拽影子。
- 插入线表示同级位置，容器高亮表示进入子级；两者不要共用含糊反馈。
- 靠近画布上下边缘自动滚动，速度与距离相关，并设置上限。
- 非法落点要显示禁用反馈，不能到松手后才报错。
- 跨容器移动前检查权限、属性兼容性和引用影响。
- 拖放完成后保持选中或短暂高亮新位置，并提供撤销。
- 必须提供键盘或菜单替代路径：上移、下移、移入、移出、移动到页面。

### Turn into

Turn into 是保持内容身份的类型转换，而不是删除后重建。

- 菜单仅列出当前块可合理转换的目标类型。
- 优先保留文本、子块、评论、引用和历史。
- 有损转换前明确提示会丢失的字段或结构；不可逆时要求确认。
- 转换后焦点保持在语义上对应的位置，不跳到页面顶部。
- 多块转换要说明结果是多个目标块、一个合并块，还是不支持。
- 数据库属性类型转换同样遵循“预览映射—指出异常—允许取消”的契约。

### 缩进与层级

- Tab / Shift+Tab 在块级选择或行首语境下执行缩进与反缩进；在文本编辑语境下需避免与真实制表输入冲突。
- 缩进必须有合法父级；第一项、锁定项或不支持子级的块应禁用并解释。
- Enter 延续当前层级；空块再次 Enter 可退出列表或回到上一级。
- 折叠父块时显示子项数量或存在性信号，避免内容“凭空消失”。
- 深层嵌套应限制视觉偏移，必要时用导引线、面包屑或聚焦模式保持可读性。
- 移动父块时默认连同全部子孙移动，并在多选或拖拽影子中表达范围。

### Mention 与 Backlink

Mention 用于建立显式引用，Backlink 用于呈现“哪些对象指向这里”。

- 输入 `@` 搜索人员、页面、日期或其他可提及对象；结果按类型分组并显示消歧信息。
- Mention 保存对象 ID，显示名称随源对象更新；对象删除或无权限时显示受控降级状态。
- 页面提及与普通超链接需视觉可区分，并支持预览来源上下文。
- 创建有效 Mention 后，在目标对象生成 Backlink；删除最后一个引用时相应移除。
- Backlink 只暴露当前用户有权查看的来源，不能通过标题或摘要泄露受限内容。
- 重复引用可聚合，但应允许展开定位到具体上下文。
- 自引用、循环引用和大规模引用网络应可用，但避免自动递归展开。

## 四、渐进披露

渐进披露不是简单“把功能藏起来”，而是让能力在用户产生意图时出现。

### 三层披露

1. **常驻层：** 内容、标题、主导航、当前任务所需的最小操作。
2. **上下文层：** hover 句柄、选中文本工具条、块菜单、属性菜单、拖拽落点。
3. **专家层：** slash 命令、快捷键、批量操作、公式、关系、自动化和高级权限。

### 触发规则

- 空页面：提供少量任务导向起点，不铺满模板卡片。
- 输入中：优先显示与光标和内容相关的动作。
- 选中块：显示块级操作；选中文本：显示行内格式操作，避免两套工具条争夺焦点。
- 首次成功后再揭示下一层能力，例如创建列表后提示可切换看板视图。
- 教学提示只出现一次或可永久关闭，不能遮挡主要操作。
- 高级功能可搜索、可被命令唤起；“隐藏”不能等于“无法发现”。
- 无权限、离线或上下文不支持时说明原因，并提供下一步，而不是让入口直接消失。

## 五、交互状态模型

每个可交互对象至少定义以下状态。状态变化应通过两种以上信号表达，不能只依赖颜色。

| 状态 | 用户意图 | 必要信号 | 机制要求 |
|---|---|---|---|
| Rest | 阅读、浏览 | 内容清晰，控件低干扰 | 隐藏低频句柄和边框；保留稳定布局，不能因控件出现而跳动 |
| Hover | 探查可操作性 | 轻背景、句柄、提示或指针变化 | 扩大命中区；不改变内容尺寸；触屏不得依赖此状态 |
| Select | 指定操作对象 | 选区、边框、计数、层级范围 | 区分文本选择、单块选择、多块选择；允许 Shift 扩展与 Esc 清除 |
| Edit | 修改内容 | 光标、输入法状态、局部工具 | 键盘输入归属明确；避免全局快捷键吞掉正文输入；保留自动保存状态 |
| Commit | 接受变更 | 内容稳定、短暂确认、保存状态 | 定义提交时机：失焦、回车、显式按钮或自动保存；支持撤销与乐观更新回滚 |
| Error | 变更未完成或无效 | 就近错误、原因、保留用户输入 | 不清空内容；提供重试、修正或复制；后台错误也需可追溯 |

### 关键状态转换

- Rest → Hover：仅增加操作暗示，不改变排版。
- Hover → Select：显示对象级边界，确保用户知道菜单或快捷键作用范围。
- Select → Edit：焦点进入内容后，块级高亮可减弱但不可丢失上下文。
- Edit → Commit：明确触发条件；异步保存可先乐观呈现，再显示同步结果。
- 任意状态 → Error：保留原值、焦点和上下文；错误恢复后回到用户之前的状态。
- Edit/Select → Rest：Esc 应按层级退出，先关浮层，再退出编辑，再清除选择。

## 六、用户感觉—信号—机制

设计评审不只检查功能是否存在，还要检查用户能否从界面信号推断系统行为。

| 目标感觉 | 用户可见信号 | 产品机制 |
|---|---|---|
| 我能立刻开始 | 空行光标、少量起点、`/` 提示 | 默认段落块、低成本创建、延迟配置 |
| 我知道自己在操作什么 | 块边界、选择数量、面包屑、插入线 | 稳定对象身份、明确作用域、选择模型 |
| 系统不会弄丢内容 | 保存状态、撤销提示、版本记录入口 | 自动保存、幂等写入、操作日志、回滚 |
| 我可以试错 | 即时预览、可撤销 toast、取消路径 | 可逆命令、软删除、转换前兼容性检查 |
| 变化符合预期 | 原位更新、焦点保持、位置高亮 | 乐观更新、滚动锚点、身份保留 |
| 界面很安静 | Rest 状态低干扰、按需出现控件 | 渐进披露、无布局跳动、上下文工具 |
| 功能找得到 | slash 搜索、统一菜单、快捷键提示 | 命令索引、同义词、可学习的信息架构 |
| 团队状态可信 | 头像、编辑位置、权限与同步信号 | 实时协同、冲突处理、权限预检 |
| 数据是一致的 | 视图来源、筛选条件、记录身份 | 单一数据源、视图投影、引用完整性 |
| 我仍掌控自己的工作方式 | 可编辑模板、个人视图、退出引导 | 模块化模板、视图作用域、可跳过 onboarding |

## 七、UI/UX 细节检查表

### 对象与层级

- [ ] 移动、复制、链接、嵌入、同步副本在文案和结果上可区分。
- [ ] 页面、记录页、数据库与视图的名称和图标表达各自身份。
- [ ] 删除视图不会删除源记录；危险操作明确影响数量和范围。
- [ ] 折叠、筛选、权限不足和真实删除有不同空状态。
- [ ] 移动页面后链接、提及、评论和历史仍然有效。
- [ ] 模板实例与模板源的同步关系被明确说明。

### 布局与视觉

- [ ] hover 控件出现时不引发布局位移。
- [ ] 命中区大于图标视觉尺寸，且不遮挡相邻操作。
- [ ] 正文、属性、元信息和系统状态具有稳定的视觉层级。
- [ ] 选中、聚焦、错误、禁用不只依靠颜色区分。
- [ ] 长标题、长属性值、窄窗口、缩放 200% 时仍可使用。
- [ ] 浮层避让视口边缘、软键盘和选中文字，不遮住当前输入。
- [ ] 动效短且服务于空间连续性，并尊重“减少动态效果”设置。

### 输入与编辑

- [ ] 点击正文进入预期光标位置，而不是只选中整个块。
- [ ] 中文、日文等输入法组合阶段不会误触 Enter、Slash 或全局快捷键。
- [ ] 粘贴纯文本、富文本、URL、表格数据时有一致且可撤销的规则。
- [ ] Enter、Shift+Enter、Tab、Backspace 在空块、列表、标题和嵌套中行为一致。
- [ ] 文本选择与块选择之间有清晰转换，不会出现“按 Delete 不知道删什么”。
- [ ] 自动保存失败不会丢失本地输入，并提供重试或复制出口。

### 菜单与命令

- [ ] 同一动作在 slash、块菜单和快捷键中的命名一致。
- [ ] 菜单按高频任务分组，危险操作与普通操作分隔。
- [ ] 菜单支持键盘导航、搜索、Esc 关闭和焦点归还。
- [ ] 禁用项说明原因；因权限缺失而隐藏的入口不会造成认知断层。
- [ ] 批量操作显示作用对象数量，并处理部分对象不兼容的情况。

### 反馈与恢复

- [ ] 创建、移动、转化、删除、分享都有即时反馈。
- [ ] 撤销恢复对象身份、位置、层级和关联，而不只是恢复可见文本。
- [ ] 长任务显示进度或阶段，且允许安全离开页面。
- [ ] 错误信息靠近问题对象，包含原因、影响和可执行下一步。
- [ ] 权限、网络、校验、冲突和系统错误使用不同文案。

### 协作与权限

- [ ] 分享前能预览访问范围和继承关系。
- [ ] 他人光标与选择不会遮挡当前编辑，颜色和姓名可辨认。
- [ ] 同时编辑冲突有确定的合并或恢复策略。
- [ ] Mention、Backlink、搜索结果和预览不会泄露无权限内容。
- [ ] 个人视图与共享视图的修改范围在执行前可见。

## 八、键盘、移动端、性能与无障碍

### 键盘

- 覆盖完整路径：创建块、选择块、多选、移动、缩进、打开菜单、执行命令、撤销和恢复。
- 快捷键按平台显示，避免把 macOS 的 Command 与 Windows 的 Ctrl 混写。
- 全局快捷键不得在正文输入、输入法组合、对话框和代码块中误触。
- 焦点顺序应符合视觉和语义顺序；浮层关闭后焦点返回触发元素。
- 提供可搜索的快捷键帮助，但不要要求用户记忆快捷键才能完成核心任务。

### 移动端

- 用底部工具条、显式“+”、更多菜单和长按替代 hover 与六点句柄依赖。
- 触控目标建议至少约 44×44 CSS 像素，并留出相邻目标间距。
- 拖拽设置长按阈值和触觉反馈，避免与滚动、文本选择冲突。
- 软键盘出现时保证当前光标、提交按钮和错误信息仍可见。
- 复杂数据库可默认提供卡片/列表摘要，横向表格作为次级模式。
- 手势必须有可见替代操作；侧滑、长按不能是唯一入口。
- 弱网和应用切后台时先保存本地草稿，并明确同步状态。

### 性能

- 首次打开优先渲染标题、首屏块和关键属性；延迟加载封面、预览和深层子块。
- 大型页面和数据库使用虚拟化，但要维护键盘焦点、搜索定位和屏幕阅读器连续性。
- 拖拽、输入和选择反馈应优先保持帧率；耗时派生计算移出主线程或延迟执行。
- 乐观更新需有幂等请求、失败回滚和重试去重，防止重复块或重复记录。
- 搜索、筛选和 slash 建议采用防抖，但本地输入与光标反馈必须即时。
- 实时协作更新应合并批次并保护当前编辑区，避免远端变化导致光标跳动。
- 设定并监控大页面、深嵌套、宽表、关系/汇总和多人编辑的性能预算。

### 无障碍

- 使用语义化结构表达标题、列表、复选框、表格、按钮、菜单和对话框。
- 每个图标按钮有可访问名称；六点句柄需说明“移动或打开块菜单”。
- 焦点样式在所有背景上清晰可见，且不被选中态覆盖。
- 状态变化通过文字或可访问通知传达；不要仅依赖颜色、位置或动画。
- 拖拽提供菜单和键盘替代，并在移动后播报新位置与层级。
- 菜单、命令面板和属性选择器遵循正确的键盘模式与焦点圈闭。
- 数据表提供行列标题与可理解的阅读顺序；虚拟化不能让辅助技术误判总量。
- 颜色对比、缩放、减少动态效果、高对比度和屏幕阅读器模式应纳入验收。

## 九、已知限制与设计边界

- **Hover 不可跨端成立：** 桌面端依靠 hover 的能力，触屏端必须有显式入口。
- **自由组合会制造复杂度：** 无限嵌套、过多属性和视图会降低理解与性能；应提供聚焦、折叠、搜索和治理能力，而不是只增加功能。
- **类型转换并非总能无损：** 不兼容字段、子结构和格式必须预告，必要时创建可恢复备份。
- **模板会漂移：** 实例自由编辑后与模板分叉是正常现象；除非有清晰版本与冲突模型，不应承诺自动同步。
- **视图容易被误认为数据：** 所有删除、共享和权限操作都要持续区分投影与源数据。
- **Backlink 受权限约束：** 反向链接不可能总是完整可见；界面应说明结果仅限当前访问范围。
- **实时协作不能消除所有冲突：** 离线、跨设备和长时间编辑仍需版本历史、冲突副本或恢复入口。
- **移动端不适合完整复刻桌面密度：** 应保留任务连续性和核心编辑能力，而不是强行塞入所有列与面板。
- **虚拟化与无障碍存在张力：** 性能优化必须同时验证焦点、朗读顺序、查找和打印/导出结果。
- **快捷键受平台、浏览器和输入法影响：** 需要冲突检测、平台适配和可替代路径。
- **权限继承越灵活越难解释：** 复杂例外需要清晰的来源说明、预览和审计，不应只用锁图标概括。
- **自动保存不等于绝对安全：** 用户仍需要同步状态、失败提示、历史版本与可导出能力来建立信任。

最终判断标准：用户不必先理解完整系统就能开始，但随着任务变复杂，可以用同一套对象和交互语法自然扩展；每次操作都能回答“对象是什么、范围在哪里、结果会怎样、失败后如何恢复”。
